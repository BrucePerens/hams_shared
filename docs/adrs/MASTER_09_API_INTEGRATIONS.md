# MASTER 09: API Integrations & Cryptography

## Status
Accepted (Consolidates ADRs 0011, 0025, 0026, 0028, 0029, 0065, 0067)

## Context & Philosophy
Communication with third-party networks, local hardware, and headless logging software requires secure, resilient pathways that cannot rely on standard browser cookies.

## Decisions & Mandates

### 1. Zero-Knowledge API Proofs (HMAC-SHA256)
Users and headless clients MUST NEVER transmit their raw API secret over the network for authenticated endpoints (where standard cookies are unavailable). Instead, they must transmit:
1. Their `user` identifier.
2. A Unix `timestamp`.
3. A highly entropic `nonce`.
4. An HMAC-SHA256 `signature` generated by hashing the `timestamp|nonce|payload` using their private API secret.
The server recalculates the signature and verifies the Nonce against a Redis cache TTL to instantly block replay attacks. Exemptions are granted only for uncontrollable third-party legacy hardware (like DDNS clients).

### 2. API Idempotency
* External clients logging data via POST MUST provide an `X-Idempotency-Key` validated against a Redis cache to prevent duplicate ingestion upon network retry.

### 3. Headless API Translation Ban
* The use of the `_()` translation wrapper around string literals within JSON responses in headless API controllers is strictly **banned**. APIs must return static, un-translated English error codes. Client-side applications are entirely responsible for localizing error strings, preventing `AttributeError` crashes when context lacks a bound user session.

### 4. Stateless HMAC-TTL Routing Tokens
* Short-lived access grants (like 1-click unsubscribe links) MUST use stateless, cryptographically signed parameters in the URL including a Unix timestamp, eliminating the need to store and garbage-collect token rows in PostgreSQL.

### 5. Ethical Crawling & Delta Checksums
* Daemons polling massive external databases MUST respect external servers by evaluating `ETag` and `Last-Modified` headers via `HEAD` requests prior to downloading.
* Downloaded payloads MUST be cryptographically hashed (SHA-256) and verified against the previous state before processing to save CPU cycles.

### 6. At-Rest Encryption for Secrets
* User credentials for third-party platforms MUST be symmetrically encrypted in PostgreSQL using the system's Fernet `HAMS_CRYPTO_KEY`.

### 7. Hardware-to-Web Airgap Bridge
* Web browsers block local network requests. Connecting the Web Shack to physical hardware requires a local relay daemon, which MUST be packaged into native, 1-click OS installers for user accessibility.
